## 🐛 4. 발견 이슈 & 해결

### 이슈 1: Docker 미설치
**증상**: Phase 1 Task 4 (Supabase 로컬 초기화)에서 실행자 subagent가 BLOCKED 반환.
**원인**: 머신에 Docker Desktop이 없음.
**해결**: 사용자가 수동으로 Docker Desktop 설치 + Supabase CLI도 brew로 설치. 재개.

### 이슈 2: Plan 파일의 테스트 버그 2건
- Phase 1 mock-seed 시드: 두 번의 독립적 `random()` 호출 → `total_inspected = good + defect + unknown` CHECK 위반 → `cross join lateral`로 수정
- `04_lots.sql` pgTAP: `prepare` 전에 불량 insert 실행으로 트랜잭션 abort → pre-prepare insert 제거

**교훈**: 자동화 에이전트가 plan을 byte-for-byte 이행해도 **plan 자체 버그**는 실행 시점에만 발견됨.

### 이슈 3: `on conflict do nothing` target 생략
**증상**: Phase 2의 alarm_rules seed 재실행 시 중복 row 삽입.
**원인**: target 컬럼 안 적으면 UNIQUE 제약 없는 컬럼에서 no-op이 아닌 그냥 INSERT.
**해결**: `alarm_rules.name`에 UNIQUE 제약 추가 + `on conflict (name) do nothing` 명시.

## 💡 5. 오늘의 교훈

1. **설계를 plan에 옮길 때 실행 가능성 검증 필수** — 에이전트는 byte-for-byte 이행하므로 plan의 SQL/코드는 반드시 해당 환경(예: Postgres random() 동작, `generate_series`의 inclusive/exclusive)을 이해해야 함.
2. **시드 SQL은 반드시 idempotent** — `on conflict`의 target은 명시적으로.
3. **외부 의존성은 프로젝트 초기에 명시** — Docker, Homebrew, Supabase CLI 등의 전제조건을 sprint 0에서 체크.
4. **로컬과 클라우드의 동일 스키마 전략이 옳았음** — 마이그레이션 하나로 두 환경 모두 커버.
